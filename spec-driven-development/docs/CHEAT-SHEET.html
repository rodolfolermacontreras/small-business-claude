Evolving Multi-Agent Framework — Cheat Sheet

One human + a fleet of AI agents shipping spec-driven features. Print, share, pin to the wall.

TL;DR — in 60 seconds

  1. Bootstrap a project (greenfield or brownfield).
  2. Open VS Code, pick the Principal Executive Manager agent.
  3. Tell it your first idea in plain English. It captures + routes.
  4. Walk the lifecycle. Approve specs at the gate. Trust the two-stage review.
  5. Run /replan after every DONE; /evolve when lessons accumulate.

Golden rule: Talk to the Executive Manager first. It either answers or routes; you never chase Principals yourself.

The lifecycle

IDEA → BACKLOG → CLARIFY → SPEC → PLAN → TASKS → IMPLEMENT → REVIEW → DONE → REPLAN

Spec sizing rule (Article VI): bug fix <3 files = no spec. Feature <5 files = lightweight spec. Feature ≥5 files = full spec. Schema/cross-cutting = full spec + ADR + human approval.

Two-stage review (Article III): spec compliance FIRST, then code quality. Different reviewers. Order is fixed.

Recommended Workflow

The recommended path from empty folder (or existing repo) to a shipped feature with full audit trail.

[H] = human approval point. GATE = quality gate. After DONE, /replan + optional /evolve close the loop back to a stronger framework for the next IDEA.

If you're not sure which archetype to pick, just open the Executive Manager first and say you're starting a new project — it will load the archetype-recommender skill and walk you through the choice.

The 10 binding articles — one line each

  1. I. Two-folder split is invariant (.github/ + spec-driven-development/)
  2. II. Single human entry point: the Executive Manager
  3. III. Two-stage review order is fixed: spec compliance, then code quality
  4. IV. Specialization over generalism: workers 1-3 files; reviewers review only
  5. V. Generic by default, specialized on demand
  6. VI. Ceremony proportional to risk (the spec sizing rule)
  7. VII. Every artifact is a file; every dispatch is logged
  8. VIII. Constitution is immutable without an ADR
  9. IX. Human holds final approval on Level-2 decisions
  10. X. Validation is a pre-implementation contract

Documented in spec-driven-development/constitution/principles.md. Amend only via /constitution — it semver-bumps and runs the propagation check.

How the agent fleet learns and evolves

1. Workers spawn generic

Software Developer Principal dispatches one of the generic worker agents (developer-general, ux-designer-general, qa-engineer-general, data-scientist-general) per task. Every dispatch is logged to fleet.db with task, worker, role, outcome.

2. Specialization is earned

A worker that excels in a domain over multiple dispatches gets a permanent identity (e.g. data-scientist-bob-forecast-1.agent.md) plus a domain-specific skill pack. Triggered through /evolve as an agent-update lesson. Old generic agents are kept; specialists join the fleet alongside.

/hire is the approved path for creating new worker roles and promoting generic workers into permanent specialists with roster, Fleet state, and ledger provenance.

3. Workers are never deleted

Article VII: history preserved. Workers get updated when /evolve ships agent-update changes (skill pack tweaks, prompt refinements, deprecation flags). The ledger records every dispatch forever.

4. Lessons captured mid-work

Any agent can load the lesson-capture skill mid-feature and append a tagged candidate to sprints/PI-{N}/lessons.md. Tags: skill-update, new-skill, agent-update, constitution-amendment, template-update, prompt-update, docs-update.

5. Lessons reviewed at /replan

After every feature DONE, /replan reviews constitution + roadmap + skills + lessons. Surfaces what should change. Doesn't auto-apply — queues for curation.

6. Lessons curated via /evolve

/evolve reads the lessons backlog and decides per candidate: SHIP (drafts the diff, references lesson ID + source feature for provenance) / DEFER (target PI) / DISCARD (with reason). Constitution-affecting lessons route through /constitution (semver + propagation).

The provenance chain: every framework change carries the lesson ID + source feature that produced it. You can always answer "why does this skill exist?" by walking back through the ledger.

Skills like archetype-recommender, lesson-capture, role-creation, constitution-sync, and tdd-gate are loaded by agents on demand to handle specific protocols. They're not user-facing slash commands but they shape every conversation.

Your first day — checklist

  1. Pick a project (start small — one week scope, one owner: you).
  2. Pick an archetype that fits.
  3. Run the bootstrap. Verify .github/ and spec-driven-development/ appear in the new repo.
  4. Open VS Code. Confirm 4 Principal agents show up in Copilot Chat agent picker.
  5. Pick the Principal Executive Manager. Greet it. Read its 3-sentence status.
  6. Tell it your first idea in plain English (1-2 sentences max).
  7. Let it route to the PM. Run /triage.
  8. Run /clarify. Answer questions; pick recommended answers when unsure.
  9. Run /spec. Read the validation contract. Approve the spec.
  10. Switch to Architect. Run /plan.
  11. Switch to Software Developer. Run /tasks, then /implement task-by-task.
  12. Run /qa. Verify validation contract is fully checked.
  13. Merge. Run /replan. File any frustrations as lessons.
  14. If lessons feel ready, run /evolve to ship the small ones now.

How to know it's working

  • GOOD The Executive Manager answers your first status question without you asking which Principal owns what.
  • GOOD Every spec has testable acceptance criteria + a validation contract before any code is written.
  • GOOD Two different reviewers sign off — spec compliance person isn't the code quality person.
  • GOOD You can answer "why was this built this way?" by opening files, not by remembering chats.
  • GOOD fleet.db shows real dispatch rows for every task you ran.
  • GOOD After 1-2 features, sprints/PI-1/lessons.md has real candidates filed.
  • WARN You skipped /clarify "to save time" and the spec ended up wrong.
  • WARN Same agent did spec compliance AND code quality review (Article III violation).
  • WARN You edited a constitution file directly without /constitution — alignment with skills/templates can drift silently.

Discipline that pays off & gotchas to avoid

DO

  • Talk to the Executive Manager first, always.
  • Approve specs at the gate. Read the validation contract.
  • Trust the spec sizing rule — don't over-ceremonialize a 2-file bug fix.
  • Run /replan after EVERY feature DONE. 60 seconds.
  • File frustrations as lessons. lesson-capture skill is fast.
  • Use --draft-only for brownfield. Always. The first time.
  • Let workers stay constrained to 1-3 files per task. Big tasks need decomposition, not bigger workers.

DON'T

  • Skip /clarify. It's the main grilling moment for a reason.
  • Edit constitution files directly. Use /constitution — it semver-bumps and propagation-scans.
  • Let one reviewer do both review stages. Article III violation.
  • Merge with unchecked validation boxes. Article X violation.
  • Add a third-party dependency to the framework CLI without an ADR (Article VII / framework guideline).
  • Discard a lesson because it's inconvenient. Discard reasons are public.
  • Try to convert a whole brownfield project at once. Adopt for ONE refactor first.
